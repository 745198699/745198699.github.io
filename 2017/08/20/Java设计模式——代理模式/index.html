Java设计模式——代理模式

发表于 2017-08-20 | 分类于 Java设计模式

背景

抽象实体 & 真是实体

现在的需求就是“代理”，操作者不能直接去调用RealSubject的方法，而是通过Proxy来访问。

静态代理

在静态代理中，Proxy要和RealSubject一样，实现Subject接口，而且在Proxy内部会持有一个真是类的实例。当外界调用Proxy接口方法是，内部会去实现对Real接口方法的调用。

代理类

使用Demo

局限性

  1. 如果Subject接口中有多个方法，那么proxy类要一一对应实现相应的方法。如果代理类对每个方法的“包装”一样，那么会有很多重复的代码。如Demo中输出bef...和after...
  2. 如果有不同接口的Subject需要代理，那么需要多种代理类

动态代理

中介InvocationHandler

实现了InvocationHandler接口中的invoke方法，内部用method.invoke调用Real的真是内容。

仔细观察可以发现，在InvocationHandler中，不在持有Subject的实例，而是Object类型的。Handler只负责实现代理的职责，在调用前后做什么样的代理工作。所以这个Handle可以处理任意类型的实体，不局限于Subject

使用Demo

代理类不再像静态代理那样“写死”，而是通过Proxy.newProxyInstance动态生成。

优势&局限性

优势：

  1. 由handler统一实现调用real前后的操作，比如Subject有两个方法，如果前后操作一样，就可以写一个Handler即可
  2. 由于代理是动态生成的，因此，如果有不用类型的接口要代理，在newProxyInstance指定即可

局限性：

  1. 如果针对不用的方法有不同的代理操作那就有点无能为力了
  2. 仍然受到interface的限制
